# Contributing to DBSnap

Thank you for your interest in contributing to DBSnap! We appreciate your help in making this project better.

## One Issue â†’ One PR

To keep changes manageable and reviewable, please strictly follow the rule: **One Issue per Pull Request**.

- Do not combine multiple tasks or fixes in a single PR.
- Ensure your PR title and description clearly reference the issue you are addressing.

## AI Usage Rules

We embrace the use of AI tools to accelerate development, but with responsibility:

- **Review Generated Code**: You are responsible for any code generated by AI. detailedly reviews logic, security, and performance implications.
- **Reference**: If an entire module or complex logic is purely AI-generated, mention it in the PR description or comments.
- **Tools**: We use Google Gemini and other advanced models. Ensure you are following the project's established patterns (e.g., using existing UI components, following the project structure).

## Commit Conventions

We follow [Conventional Commits](https://www.conventionalcommits.org/) to create a readable history.

Format: `<type>(<scope>): <subject>`

### Types

- **feat**: A new feature
- **fix**: A bug fix
- **docs**: Documentation only changes
- **style**: Changes that do not affect the meaning of the code (white-space, formatting, etc)
- **refactor**: A code change that neither fixes a bug nor adds a feature
- **perf**: A code change that improves performance
- **test**: Adding missing tests or correcting existing tests
- **chore**: Changes to the build process or auxiliary tools and libraries such as documentation generation

### Example

```
feat(auth): add google oauth login support
fix(api): handle missing user profile in response
docs: update readme with setup instructions
```
